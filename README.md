# ad-realtime-data-pipeline

광고 노출, 클릭, 전환 로그를 실시간으로 수집하고 정제한 뒤 캠페인 성과 지표와 광고주 리포트 Mart를 만드는 로컬 데이터 파이프라인 포트폴리오 프로젝트입니다.

이 프로젝트는 실무의 `application log -> collector -> Kafka -> stream processing -> warehouse/mart` 흐름을 작게 재현합니다. 이벤트 생성기는 Kafka를 직접 알지 못하고 JSONL 로그 파일만 생성하며, Kafka 적재는 별도의 collector가 담당합니다.

## Architecture

```text
src/generator/event_generator.py
-> data/landing/*.jsonl
-> src/collector/log_collector.py
-> Kafka ad-events-raw
-> Java Flink CleanAdEventsJob
-> PostgreSQL Bronze/Silver + Kafka ad-events-clean
-> Java Flink CampaignMetricsJob
-> ClickHouse gold_campaign_metrics_minutely + Kafka ad-metrics-realtime
-> Airflow daily_ad_report_dag
-> ClickHouse hourly/daily/advertiser marts
```

## Components

`event_generator`

- Generates JSONL application logs under `data/landing/`.
- Creates `impression`, `click`, and `conversion` events.
- Can intentionally mix invalid, duplicate, and late events.
- Does not send data to Kafka.

`log_collector`

- Reads `data/landing/*.jsonl`.
- Sends each valid JSON line to Kafka topic `ad-events-raw`.
- Uses `user_id` as the Kafka message key.
- Moves processed files to `data/archive/`.
- Writes malformed records or records without `user_id` to `data/bad/`.

`CleanAdEventsJob`

- Consumes Kafka `ad-events-raw`.
- Parses JSON into `AdEvent`.
- Validates required fields, event type, money fields, and conversion fields.
- Routes invalid events to PostgreSQL `silver_invalid_events`.
- Applies event-time watermark with 10 minute bounded out-of-orderness.
- Routes late events to PostgreSQL `silver_late_events`.
- Deduplicates by `event_id`.
- Deduplicates conversion events by `conversion_id`.
- Stores clean events in PostgreSQL `silver_ad_events`.
- Publishes deduplicated clean events to Kafka `ad-events-clean`.

`CampaignMetricsJob`

- Consumes Kafka `ad-events-clean`.
- Uses `event_time` based 1 minute tumbling windows.
- Aggregates by `campaign_id` and `advertiser_id`.
- Calculates impressions, clicks, conversions, CTR, CVR, cost, revenue, and ROAS.
- Stores results in ClickHouse `gold_campaign_metrics_minutely`.
- Optionally publishes metrics to Kafka `ad-metrics-realtime`.

`daily_ad_report_dag`

- Runs in Airflow.
- Rebuilds hourly, daily, and advertiser daily marts in ClickHouse.
- Uses delete-insert by `metric_date` so the DAG can be rerun.
- Leaves a replay/recompute extension point for PostgreSQL `silver_late_events`.

## Kafka Topics

| Topic | Producer | Consumer | Purpose |
| --- | --- | --- | --- |
| `ad-events-raw` | `log_collector` | `CleanAdEventsJob` | Raw application log events |
| `ad-events-invalid` | Reserved | Reserved | Optional invalid event topic |
| `ad-events-deduplicated` | Reserved | Reserved | Optional dedup topic |
| `ad-events-clean` | `CleanAdEventsJob` | `CampaignMetricsJob` | Valid, non-late, deduplicated events |
| `ad-metrics-realtime` | `CampaignMetricsJob` | Optional clients | Real-time campaign metrics |

Kafka partition key is `user_id` for event topics.

## Storage

PostgreSQL is used as Bronze/Silver operational storage.

| Table | Role |
| --- | --- |
| `bronze_ad_events` | Raw event payload storage |
| `silver_ad_events` | Valid clean events |
| `silver_invalid_events` | Schema or data quality failures |
| `silver_duplicate_events` | Duplicate `event_id` or `conversion_id` events |
| `silver_late_events` | Events later than the 10 minute watermark |

ClickHouse is used as Gold Mart / serving storage.

| Table | Role |
| --- | --- |
| `gold_campaign_metrics_minutely` | Real-time 1 minute campaign metrics |
| `gold_campaign_metrics_hourly` | Hourly campaign report |
| `gold_campaign_metrics_daily` | Daily campaign report |
| `gold_advertiser_mart_daily` | Daily advertiser mart |

## Metric Rules

Campaign metrics are calculated per `campaign_id`, `advertiser_id`, and time window.

- `impressions`: count of impression events
- `clicks`: count of click events
- `conversions`: count of conversion events
- `ctr = clicks / impressions`
- `cvr = conversions / clicks`
- `cost = sum(cost)`
- `revenue = sum(revenue)`
- `roas = revenue / cost`

Division by zero is always safe:

- If `impressions = 0`, CTR is `0`.
- If `clicks = 0`, CVR is `0`.
- If `cost = 0`, ROAS is `0`.

## Duplicate and Late Event Policy

Duplicate policy:

- `event_id` is the primary deduplication key.
- Conversion events are also deduplicated by `conversion_id`.
- Duplicate events are not discarded silently.
- Duplicate events are stored in PostgreSQL `silver_duplicate_events`.
- Only deduplicated events are published to `ad-events-clean`.

Late event policy:

- Stream processing uses `event_time`.
- Watermark delay is 10 minutes.
- Events older than the current watermark are routed to PostgreSQL `silver_late_events`.
- Late events are not included in real-time metrics.
- Batch reporting is designed so late events can be replayed into corrected marts later.

## Local Setup

Prerequisites:

- Docker Compose
- Python 3.11 or later
- Java 17
- Maven, or Docker access for the Maven container fallback used by `scripts/e2e_run.sh`

Create local environment variables:

```bash
cp .env.example .env
```

Install Python dependencies:

```bash
python3 -m pip install -r requirements.txt
```

Start infrastructure:

```bash
docker compose up -d
```

Service UIs:

- Kafka UI: `http://localhost:8080`
- Flink UI: `http://localhost:8081`
- Airflow UI: `http://localhost:8082`
- Airflow login: `admin / admin`

Host PostgreSQL port is `5433`; containers use `postgres:5432` internally.

## End-to-End Run

Run the full local pipeline:

```bash
./scripts/e2e_run.sh
```

Useful options:

```bash
EVENT_COUNT=500 EVENT_SEED=42 WAIT_SECONDS=30 ./scripts/e2e_run.sh
```

Check current state:

```bash
./scripts/e2e_status.sh
```

The E2E script performs:

1. Docker Compose startup
2. Kafka topic check
3. PostgreSQL and ClickHouse table check
4. Flink Maven build
5. `CleanAdEventsJob` submission
6. `CampaignMetricsJob` submission
7. JSONL event generation
8. JSONL log collection into Kafka
9. PostgreSQL Silver row count check
10. ClickHouse minutely metric check
11. Airflow DAG trigger
12. Final ClickHouse mart check

Build Flink jobs:

```bash
cd flink-jobs
mvn package
cd ..
```

Submit cleaning job:

```bash
docker compose exec flink-jobmanager flink run \
  -c com.example.adpipeline.jobs.CleanAdEventsJob \
  /opt/flink/usrlib/ad-pipeline/target/ad-realtime-data-pipeline-flink-jobs-0.1.0.jar
```

Submit metrics job:

```bash
docker compose exec flink-jobmanager flink run \
  -c com.example.adpipeline.jobs.CampaignMetricsJob \
  /opt/flink/usrlib/ad-pipeline/target/ad-realtime-data-pipeline-flink-jobs-0.1.0.jar
```

```bash
python3 -m src.generator.event_generator --count 1000 --seed 42
```

## Troubleshooting

Docker CLI is not available:

```text
The command 'docker' could not be found
```

Enable Docker Desktop WSL integration or run commands in an environment with Docker Compose.

Maven is not available:

```text
mvn: command not found
```

Install Maven and make sure Java 17 is available.

PostgreSQL or ClickHouse tables are missing after SQL changes:

Docker init scripts run only on first volume creation. For a fresh local reset:

```bash
docker compose down -v
docker compose up -d
```

Airflow DAG is not visible:

```bash
docker compose exec airflow-scheduler airflow dags list | grep daily_ad_report_dag
```

If it is still missing, check scheduler logs:

```bash
docker compose logs airflow-scheduler
```

Flink jobs are not processing:

```bash
docker compose exec flink-jobmanager flink list
docker compose logs flink-taskmanager
```

Kafka has no raw events:

```bash
ls data/landing
python3 -m src.generator.event_generator --count 100
python3 -m src.collector.log_collector
```

## Portfolio Decisions

1. Event generation and Kafka producer responsibilities are separated.

   `event_generator` writes JSONL logs only. `log_collector` is the only Python component that sends records to Kafka. This mirrors the operational pattern of application logs being collected by a separate agent or collector.

2. PostgreSQL and ClickHouse have different responsibilities.

   PostgreSQL stores Bronze/Silver operational data for traceability and quality analysis. ClickHouse stores Gold marts for analytical serving and fast report queries.

3. Java Flink is used for stream processing.

   The Flink jobs implement Kafka consumption, event-time watermarking, validation, deduplication, late event routing, window aggregation, and JDBC sinks with a type-safe Java codebase.

4. `event_time` and `ingestion_time` are separated.

   This makes late event behavior explicit and allows the project to explain the difference between real-time metrics and corrected batch reports.

5. Duplicate and late events are retained.

   Invalid, duplicate, and late data is routed to dedicated Silver tables instead of being dropped. That improves operational debugging and data quality visibility.

6. Real-time metrics and batch reports are separated.

   Real-time Flink output is optimized for speed. Airflow batch reporting is rerunnable and designed for correction/recalculation.
